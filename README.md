La metodología en cascada es especialmente beneficiosa cuando el cliente tiene requerimientos claros y fijos,
y cuando el proyecto tiene un alcance manejable. Su enfoque estructurado y su énfasis en la documentación hacen
que sea una opción sólida para asegurar que todas las fases del proyecto se realicen de manera ordenada y completa, 
minimizando riesgos y facilitando la gestión del proyecto.
La distribución de roles es fundamental para el éxito de este método, asegurando que cada miembro del equipo tenga 
responsabilidades definidas. 

## 1.Recopilación de Requisitos:
En esta fase se identifican y documentan los requisitos del sistema a desarrollar. 
Definir claramente lo que se espera del sistema, evitando malentendidos más adelante.
Producto final: Documento de Requisitos de Software (DRS).
## 2. Diseño del Sistema
Esta etapa implica la traducción de los requisitos recolectados en un diseño técnico del sistema.
Proporcionar una guía clara sobre cómo se implementará el software.
Producto final: Documento de Diseño del Sistema (DDS), incluyendo diagramas y especificaciones técnicas.
## 3. Implementación (Desarrollo)
Durante esta fase, el diseño se convierte en código. 
Construir el sistema funcional que cumpla con los requisitos definidos.
Producto final: Código fuente del software y, a veces, documentación técnica relacionada.
## 4. Pruebas (Testing)
Aquí se realiza la verificación y validación del software desarrollado. 
Detectar errores y defectos en el software antes de su implementación.
Producto final: Informe de Pruebas que documenta los resultados y defectos encontrados.
## 5. Implementación y Despliegue
En esta etapa, el software se entrega al cliente y se implementa en el entorno de producción.  
Asegurar que el sistema esté operativo y disponible para su uso.
Producto final: Software desplegado, manuales de usuario y, en ocasiones, documentación de soporte.
## 6. Mantenimiento
Esta fase abarca cualquier modificación, corrección de errores o actualización que sea necesaria después de la implementación.
Garantizar la operatividad continua del software y realizar mejoras según el feedback de los usuarios.
Producto final: Actualizaciones de software, parches y documentación actualizada.

## Planteamiento del proyecto
El cliente quiere una aplicación sencilla llamada “TaskMaster”, diseñada para gestionar tareas personales.
La idea es permitir que el usuario pueda:
-Crear tareas
-Editarlas
-Marcar tareas como completadas
-Eliminarlas
-Organizar tareas por prioridad
El cliente lo describió como:
“Quiero una aplicación fácil de usar, visual y que me permita organizar mi día sin complicaciones.”

## Etapas del ciclo de vida del proyecto
1. Análisis de requisitos:
-Reunión inicial con el cliente
-Identificación de funciones necesarias
-Separación entre requisitos funcionales y no funcionales 
2. Diseño del sistema
Incluye:
-Diseño de la interfaz
-Arquitectura básica
-Diagrama de flujo
-Estructura del código
-Base de datos local 
3. Implementación:
-Programación del proyecto en Python 
-Creación de módulos
-Diseño visual sencillo
-Validación de entradas
4. Verificación:
-Pruebas unitarias
-Pruebas de interfaz
-Comprobación del funcionamiento de cada botón
-Pruebas de usuario: comprobar que es intuitivo
5. Mantenimiento:
-Corrección de errores futuros
-Actualización según el feedback del usuario
-Posible ampliación: sincronización en la nube, notificaciones, etc.

## Roles del equipo
Aunque se realiza en grupo, los roles serían:
Analista → persona 1
Diseñador →persona 1
Programador → persona 2
Tester →persona 2

## Requisitos del proyecto
Requisitos funcionales:
1. Permitir crear nuevas tareas.
2. Guardar título y descripción de cada tarea.
3. Cambiar prioridad (alta, media, baja).
4. Marcar tareas como completadas.
5. Editar tareas existentes.
6. Eliminar tareas.
7. Mostrar todas las tareas en una lista organizada.

## Requisitos no funcionales 
1. Interfaz sencilla e intuitiva.
2. Tiempo de respuesta rápido.
3. Debe funcionar sin conexión.
4. Código organizado y modular.
5. Documentación interna del proyecto.
6. Bajo consumo de recursos.

## Resumen de cómo se llevaría a cabo el proyecto
1. Reunión inicial para definir bien la aplicación.
2. Recopilación y documentación de requisitos funcionales y no funcionales.
3. Diseño completo de pantallas, flujo de acciones y arquitectura.
4. Programación de la aplicación siguiendo lo diseñado.
5. Pruebas exhaustivas para asegurar que todo funciona correctamente.
6. Entrega final al cliente y creación del PDF para Moodle.
7. Mantenimiento según errores o mejoras solicitadas posteriormente.
